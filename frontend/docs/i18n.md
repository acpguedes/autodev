# Internationalization (i18n) — E15-S4

The Web UI ships a lightweight, dependency-free i18n layer covering the
default `en` locale and `pt-BR`. It lives entirely in `frontend/lib/i18n/`
and `frontend/locales/`, with no runtime i18n library added as a
dependency — the App Router, TypeScript, and `localStorage` are enough.

## File layout

```
frontend/locales/
  en.json        # default locale, source of truth for available keys
  pt-BR.json      # translated dictionary, structurally identical to en.json

frontend/lib/i18n/
  locales.ts       # DEFAULT_LOCALE, SUPPORTED_LOCALES, LOCALE_LABELS, Locale type
  dictionaries.ts   # Dictionary type, translate(), resolvePath()/interpolate() helpers
  I18nProvider.tsx  # "use client" context provider + useTranslations() hook
  index.ts          # barrel re-export
```

## Usage

```tsx
"use client";
import { useTranslations } from "@/lib/i18n";

function Example() {
  const { t, locale, setLocale } = useTranslations();
  return <p>{t("chat.sessionsCount", { count: 3 })}</p>;
}
```

- `t(key, vars?)` resolves a dot-path key (e.g. `"chat.errors.sendMessage"`)
  against the active locale's dictionary. `{{placeholder}}` tokens in the
  string are interpolated from `vars`.
- `locale` / `setLocale(next)` read and update the active locale. The
  choice is persisted to `localStorage` under the `autodev.locale` key and
  restored on the next visit.
- `document.documentElement.lang` is kept in sync with the active locale
  (WCAG 2.2 SC 3.1.1).

## Adding a new string

1. Add the key to `frontend/locales/en.json`.
2. Add the matching key, translated, to `frontend/locales/pt-BR.json` in
   the same nested position.
3. Use `t("your.new.key")` at the call site.

Step 2 is enforced at compile time, not just by convention: `dictionaries.ts`
declares `type Dictionary = typeof en` and then assigns the imported
`pt-BR.json` to a `const` of that type. If `pt-BR.json` is missing a key (or
has an extra/mismatched one), the TypeScript build fails immediately —
there is no way to ship a UI string that silently falls back to raw-key
text for a supported locale.

## Fallback behavior

`translate()` still defines a runtime fallback chain — requested locale →
`en` → the raw key itself — as a defensive net for dynamically-constructed
keys that bypass the compile-time check. Each fallback step logs a
`console.warn` in the browser (never server-side, never in tests) so a
regression is visible in the console during development. In practice this
path should be unreachable for any statically-known key, since those are
already guaranteed complete by the `Dictionary` typing described above.

## Adding a new locale

1. Create `frontend/locales/<locale>.json` with the same shape as `en.json`.
2. Add the locale to `SUPPORTED_LOCALES` and `LOCALE_LABELS` in
   `frontend/lib/i18n/locales.ts`.
3. Register the dictionary in `dictionaries.ts`'s `dictionaries` map, with
   a `const <locale>Dictionary: Dictionary = <locale>;` assignment to get
   the same compile-time completeness guarantee as `pt-BR`.

## Linting

`eslint-plugin-i18next`'s `no-literal-string` rule is enabled globally as a
warning, and escalated to an error via `.eslintrc.json` `overrides` for the
two files this story migrated (`app/page.tsx`,
`components/ExecutionConsolePanel.tsx`), so new hardcoded strings in those
files fail `next lint`. Story-book files, tests, and e2e specs are exempt.
Other, not-yet-migrated shell/nav surfaces remain at the global `warn`
level until they are addressed in a follow-up story.

## Scope of this story (E15-S4)

Externalized: `app/page.tsx` (chat workspace copy, error messages, plan
labels) and `components/ExecutionConsolePanel.tsx` (execution console
copy). `components/ChatLayout.tsx` was retired by E15-S2/S3 and has no
hardcoded copy to migrate. Dynamic, backend-sourced values (run status
enums, task titles/descriptions, plan step labels) are intentionally left
untranslated — they are runtime data, not UI strings.
