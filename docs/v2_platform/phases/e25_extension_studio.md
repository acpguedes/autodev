# E25 — Extension Studio: AI-Assisted Agent/Skill/Plugin Development

**Wave:** v2.1 — Spec & Harness (can start after E20 + E23-S1/S2; S4's
marketplace half is gated on E13).
**Status:** Not started · **Stories:** 0/4 complete
**Depends on:** E1 (SDK/scaffolding/contract tests), E6 (skills), E12-S2
(contract-test gates), E20 (extension specs), E23 (harness runs the build loop)
**Enables:** community extension authoring at platform level; feeds E13
(marketplace supply)
**Canonical source:** `docs/architecture/v2_platform_reference.md` §22.8,
§18.7.17; RFC-007

## Objective

Let users build the platform's own extensions — agents, skills, plugins, flows,
evals — *inside* the platform, assisted by AI and governed by the same
spec/harness machinery: scaffolding through the existing SDK exposed via `/v2`
and the UI, spec-driven authoring where an extension spec compiles to manifest
+ code + contract tests generated by a harness run (dogfooding E23), activation
gated on contract tests/evals/sandboxed test-runs, and a publish path into the
local registry today and the E13 marketplace when it lands.

## Key result

A user describes the extension they need (e.g. "a skill that lints SQL
migrations"); the Studio produces an extension spec, a harness run generates
the manifest, implementation, and contract tests; the extension only activates
after its gates pass in the sandbox; and one action publishes it to the local
registry — the whole path driven from the UI over `/v2` only.

## Prior art (condensed)

Claude Code skills/plugins (SKILL.md progressive disclosure, scaffolded
extensibility), Goose recipes, Tessl tiles (composable capability packs), and
the platform's own E1-S4 `sdk new` + contract-test harness — E25 is primarily
a productization of E1/E12 seams behind an AI-assisted authoring loop. Sources
in RFC-007.

## Stories

### E25-S1 — Scaffolding service & templates

Subtasks:
- `E25-S1-T1`: expose the E1-S4 `sdk new` scaffolding through `/v2` (create a draft extension workspace server-side) for every manifest kind: plugin, agent, skill, flow, eval.
- `E25-S1-T2`: template catalog — per-kind starter templates (deterministic skill, llm-assisted skill, agent with tools, validation gate, evaluator) mirroring `templates/manifests/`.
- `E25-S1-T3`: Studio screen — pick kind + template, name/namespace validation, draft workspace listing (extends the E17 extensions hub).

| Criterion | Detail |
| --- | --- |
| Functional | Creating a draft via UI/API yields the same layout as the CLI `sdk new`; invalid ids/namespaces rejected at creation; drafts are tenant-scoped and listable |
| Non-functional | No divergence between CLI and API scaffolds (single scaffolding implementation); API-first (§2.13) |
| DoR (specific) | E1-S4 scaffolding reviewed; extensions hub (E17-S5) patterns reviewed |
| DoD (specific) | Scaffold parity test (CLI vs API); UI e2e create-draft test; docs |
| Dependencies | E1-S4, E9-S1, E17-S5 |

### E25-S2 — Spec-driven extension authoring (dogfooding)

Subtasks:
- `E25-S2-T1`: extension-spec profile — an E20 spec whose requirements describe the extension's behavior (EARS clauses over its typed IO) and whose acceptance scenarios become its contract-test cases.
- `E25-S2-T2`: builder harness — a packaged E23 harness (`autodev/harness-extension-builder`) that iterates: generate/refine manifest + implementation → run contract tests + acceptance scenarios in the sandbox → repeat until gates pass or a typed stop state.
- `E25-S2-T3`: authoring UX — clarify loop over the extension spec (reusing E24-S2 patterns), live iteration timeline, diff review of generated code before activation.

| Criterion | Detail |
| --- | --- |
| Functional | From an approved extension spec, the builder harness produces an extension whose contract tests and acceptance scenarios pass in the sandbox; a spec the builder cannot satisfy terminates in a typed stop state with the failing gates listed — never a silent partial extension |
| Non-functional | Generated code carries generation stamps; the builder itself is an ordinary harness (budgeted, traced, resumable) — no privileged path |
| DoR (specific) | E20-S1 (spec profile), E23-S1/S2/S3 available; E22-S1 acceptance compiler available for scenario execution |
| DoD (specific) | End-to-end build of a reference skill from spec; stop-state test; docs |
| Dependencies | E20, E22-S1, E23, E24-S2 (patterns) |

### E25-S3 — Activation gates

Subtasks:
- `E25-S3-T1`: gate pipeline for any Studio-built extension: schema validation → E12-S2 contract tests → sandboxed test-run (E14-S4) → optional evals (E22-S2) — all green before the Plugin Host will enable it.
- `E25-S3-T2`: permission review step — the extension's requested permissions (deny-by-default, E1-S3) surfaced for explicit human approval before first enable.
- `E25-S3-T3`: gate results persisted and attached to the extension version (auditable "why was this allowed to activate").

| Criterion | Detail |
| --- | --- |
| Functional | An extension failing any gate cannot be enabled (fail closed); permission grants are explicit, recorded, and revocable; gate evidence is queryable per extension version |
| Non-functional | Gate pipeline reuses E12/E14 infrastructure (no second test runner); no network in gate sandbox by default |
| DoR (specific) | E12-S2 contract-test harness available; E14-S4 runners available |
| DoD (specific) | Fail-closed tests per gate; permission-review e2e test; docs |
| Dependencies | E25-S2, E12-S2, E14-S4, E1-S3 |

### E25-S4 — Publish path

Subtasks:
- `E25-S4-T1`: publish a gated extension version to the local (tenant) registry with immutable versioning — installable in other projects of the same instance.
- `E25-S4-T2`: export bundle — a portable plugin package (manifest + code + tests + gate evidence) matching what E13's marketplace publish flow will consume; signing/SBOM deferred to E13 integration.
- `E25-S4-T3`: Studio publish UX — version notes, gate-evidence summary, install-from-registry flow in the extensions hub.

| Criterion | Detail |
| --- | --- |
| Functional | A published extension is installable and enableable in another project on the same instance with its gate evidence visible; the export bundle round-trips through the Plugin Host install path |
| Non-functional | Published versions immutable; publish requires all E25-S3 gates green (enforced server-side) |
| DoR (specific) | E25-S3 available; E13 publish contract reviewed (align the bundle format even before E13 lands) |
| DoD (specific) | Publish/install round-trip test; docs |
| Dependencies | E25-S3, E1-S5; E13 (marketplace half only) |

## v1/v2 precursor / starting point

- E1-S4 already ships `sdk new` scaffolding, a runnable example plugin, and the
  contract-test harness; E1-S5/E16-S4 give the registry and the unified
  extensions catalog + enable/disable APIs; E17-S5 gives the extensions hub
  screen. E25 is largely the productization of those seams — the genuinely new
  pieces are the extension-spec profile, the builder harness, and the gate
  pipeline assembly.
- Nothing today generates extension code from a spec or gates activation on
  sandboxed test evidence.
